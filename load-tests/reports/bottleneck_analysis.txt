🔴 [CRITICAL #1] SYNCHRONOUS SUPABASE CLIENT BLOCKS ASYNC EVENT LOOP
   File: core/database.py:1-12 + all 22 route files
   Issue: supabase-py Client uses synchronous httpx internally. Every
          .execute() call blocks the asyncio event loop while waiting
          for the network response. With N concurrent users, N-1 users
          are frozen while one DB call completes.
   Evidence: get_supabase() returns sync Client; run_query() wraps in
             asyncio.to_thread() — this mitigates but does NOT eliminate
             the problem (thread pool is still finite at OS level).
   Estimated impact: With 4 uvicorn workers (current Procfile), the
             effective DB concurrency ceiling is ~20-40 requests before
             thread pool saturation causes queue buildup.
   Fix:   Switch to AsyncClient when supabase-py stable async API is
          available. Short-term: ensure run_query() is used everywhere
          (audit shows it IS used correctly in all routes — good).
          Long-term: migrate to httpx.AsyncClient direct PostgREST calls.
   Status: PARTIALLY MITIGATED — run_query() offloads to thread pool.
           True fix requires async Supabase client.
